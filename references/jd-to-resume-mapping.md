# JD 解析与简历映射规则

## JD 解析四维框架

拿到 JD 后，按以下四个维度依次分析，结论直接影响简历内容、排序和语气。

---

### 维度1：职位性质

确定以下两个坐标：

**招聘类型**：

- 实习（Intern）：学生身份，时间有限，强调学习速度和配合能力
- 校招（Campus Hire）：应届生，强调学习曲线、项目结果
- 社招（Social Hire）：有工作经验，不在本Skill覆盖范围

**职能方向**：

- 技术类（研发/数据/算法）：代码、建模、系统设计
- 产品类：需求分析、原型、用户洞察、优先级
- 运营类：活动策划、数据运营、增长、用户沟通
- 研究类：文献、实验、报告、量化分析
- 综合/管培：全局视角、学习速度、跨职能

---

### 维度2：岗位要求（硬条件）

硬条件 = JD 里明确写"必须""要求""需要"的内容。

常见硬条件类型：

- 学历要求：本科/硕士、特定专业
- 技术栈：Python/SQL/Java/C++
- 工具：Tableau/Excel/SPSS/Figma
- 语言：英语CET-4/CET-6/雅思/托福
- 证书：CFA/CPA/教师资格证

**映射规则**：

- 用户满足 → 必须出现在技能栏或教育背景里，不可埋没
- 用户不满足 → 不要在简历里假装满足，不强行堆词

---

### 维度3：加分项（软偏好）

JD 里"优先""加分""有XX经验者优先"的内容。

**映射规则**：

- 用户有对应经历 → 主动前置，给该经历更多篇幅
- 用户没有 → 跳过，不强行编造

---

### 维度4：企业调性

企业类型决定整份简历的语气、重点和格式（见 resume-writing-rules.md 场景语气节）。

从以下信号判断：

- 公司名称（国字头 vs 科技公司 vs 外资品牌）
- JD 语言风格（官方严肃 vs 直接量化 vs 英文为主）
- 工作地点（北上深广 vs 省会城市 vs 体制内城市）

---

## JD 元素 → 简历动作 映射表


| JD 信号        | 简历动作                               |
| ------------ | ---------------------------------- |
| 要求Python/SQL | 技能栏首行；有项目用到的在项目里用动词"利用Python..."体现 |
| 要求数据分析       | 把有数字/量化的项目排前；bullet强调结果数字          |
| 要求沟通/协作      | 校园经历选有跨团队、组织活动的；bullet里体现"协调""对接"  |
| 要求产品思维       | 优先保留有需求分析、原型、用户调研的项目               |
| "积极主动"       | 加一条"独立发起"或"主动提出"的具体事件              |
| 研究/分析岗       | GPA前置；科研项目详写；方法论（回归/访谈/内容分析）显化     |
| 国企/选调        | 学生干部职务前置；政治面貌写上；集体荣誉突出             |
| 互联网          | 每条bullet必须有量化；删掉没有结果的流水账           |
| 外企           | 强调英语水平；如有海外/国际项目经历，提前排序            |

## 同义归并规则

以下概念在不同JD里表达不同，但指向同一能力。
agent 做映射时要识别这些同义关系，不要因为词不完全一样就漏匹配。

### 数据分析能力

等价表述集合：

- 数据分析 / 量化分析 / 数据处理
- Python能力 / SQL能力 / Pandas/NumPy
- 统计分析 / 建模经验 / 机器学习基础
- 数据可视化 / Tableau / Power BI / Excel高级

如果JD说"数据分析"，简历里有Python项目→应该体现，不要因为JD没写Python就跳过。

### 沟通协作能力

等价表述集合：

- 沟通能力 / 表达能力
- 跨部门协作 / 跨团队合作
- 向上管理 / 项目汇报
- 组织协调 / 资源调配
- 客户沟通 / 用户访谈

### 产品/运营能力

等价表述集合：

- 产品思维 / 用户视角 / 同理心
- 需求分析 / 需求挖掘 / 用户调研
- 原型设计 / Axure / Figma / 墨刀
- 活动策划 / 运营活动 / 增长实验
- 数据驱动 / AB测试 / 用户分层

### 研究/学术能力

等价表述集合：

- 文献综述 / 文献调研 / 文献检索
- 实验设计 / 研究方法 / 定量研究 / 定性研究
- 论文写作 / 报告撰写 / 研究报告
- 数据录入 / 问卷设计 / 访谈提纲

## 反面示例（映射失败的典型错误）

### 错误1：硬技能被埋在末尾

**JD 要求**：熟练使用 Python，有数据处理经验

**错误做法**：

```
技能特长：
• 语言：英语六级
• 软件：Office、Photoshop
• 其他：Python（了解）
```

**正确做法**：

```
技能特长：
• 编程：Python（Pandas/NumPy/matplotlib，有独立项目经验）
• 数据：SQL（增删改查），Excel（数据透视、VLOOKUP）
• 语言：英语六级（XXX分）
```

Python 提到第一位，标注实际掌握程度，附上项目佐证。

---

### 错误2：忽略加分项

**JD 要求**：有竞赛经历者优先

**错误做法**：把相关竞赛埋在自我评价里一句带过，或根本没写

**正确做法**：

- 在奖项/荣誉或项目经历里完整列出竞赛名称、级别、结果
- 如果是重要加分项，可在该条目前加粗标注获奖级别

---

### 错误3：调性不匹配

**JD 来自**：某市国资委选调生

**错误做法**：

```
主导搭建了社团增长运营体系，通过数据驱动实现了用户量3倍提升。
```

问题：语气偏互联网，"增长运营体系""数据驱动""用户量"是互联网术语，国企HR读起来违和。

**正确做法**：

```
担任社团副会长，统筹组织5场年度活动，协调80名成员完成分工，
活动参与人数较上一年度增长超3倍。
```

语气改为"组织、统筹、协调"，保留数字，去掉互联网术语。

---

### 错误4：同义词漏匹配

**JD 要求**：有跨部门沟通协作经验

**错误做法**：候选人有"协调外联工作"的经历，agent 没识别出来是跨部门协作，没有提前。

**正确做法**：识别"协调外联"≈"跨部门沟通"，将该条经历前置，并改写为：

```
负责对接3个合作方，统筹推进联合活动全流程，在资源冲突时独立协调解决方案。
```

## 优先级决策树

当简历内容过多需要取舍时，按以下顺序决定保留哪些：

1. 满足硬条件的经历/技能 → 必须保留
2. 满足加分项的经历 → 优先保留
3. 有量化结果的经历 → 优先于无量化的
4. 与目标岗位职能方向最近的 → 优先
5. 时间最近的 → 相同权重下优先
6. 纯凑版面的经历 → 删掉

## 资料补充

### 项目质量筛选标准

多个经历中如何挑出最值得写的，按以下条件优先级排序：

1. **时效性**：2年内参与的经历优先（过早的经历与当前能力无关联）
2. **持续性**：持续3个月以上的项目优先（说明有深度参与，不是打酱油）
3. **来源含金量**：大厂实习 > 小厂实习；国家自然科学基金项目 > 学校科研训练计划项目
4. **结果可见性**：有可核实成果（发布、奖项、被采纳）的 > 无成果的

### 多版本差异化策略

同一个人投不同岗位时，简历不能千篇一律。保留多个版本，每个版本的差异化策略：

**策略A：相同性质+不同工作内容**

- 岗位方向相同（如都是产品），但不同经历突出不同职能侧重（用研 vs 运营 vs B端）
- 根据JD重新排列经历顺序，把最匹配的放前面

**策略B：技能相同+不同应用领域**

- 同一技术能力（如数据分析）对应不同行业场景（金融 vs 互联网 vs 政务）
- 技能栏保持不变，经历描述换场景关键词

**策略C：A/B版本分工**

- 版本A：突出专业技能（给技术类岗位）
- 版本B：突出个人特质和沟通协作（给产品/运营/综合管培）
- 两个版本选取的经历可以不同，同一经历的bullet侧重点不同

### 多项目简历内的排序规则

当简历里有多条经历时，排序不是随机的：

- **首选按与目标岗位的相关度排**：最相关的经历放最前，而非时间最近的
- **相关度相近时，按时间从晚到早排**
- **不要把所有经历混排**：实习经历、项目经历、校园经历应分区块，块内再排序

### 不一份简历一投到底

这是反面原则：**绝不维护一份"万能简历"对所有岗位无差别投递**。

具体后果：万能简历必然对任何岗位都不是最优版本——既不能最大化匹配度，也会因为装了所有内容而超页或稀释重点。

正确做法：按岗位类别维护2–3个版本，每个版本针对性裁剪，投前确认版本正确。
