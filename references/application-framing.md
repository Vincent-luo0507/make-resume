# 简历生成四阶段：Application Framing

这是 generate phase 的**质量核心文档**。读完 profile + JD，**不要**直接开始写简历——按下面四个阶段走，最后做六项 review，不通过就回滚重来。

## 核心目的（唯一目标）

> **让 HR 看到简历时直观清晰地认识到：你很强，你就是这个岗位所需要的人。**

如果生成出来的简历不能让一个不了解候选人的 HR 在 20 秒内回答出「这个人为什么适合这个岗位」，就是失败。

## 五大原则（所有判断的基准）

| 原则 | 含义 | 可量化标准 |
|-----|------|----------|
| **人岗匹配** | 你是他们要的人 | 简历里能直接找到 JD 的核心关键词、命题、硬条件 |
| **内容充实** | 你很强 | 至少 3 段能体现岗位能力且能展开 ≥3 行的经历；至少 1 段与专业技能密切相关 |
| **扬长避短、突出自我** | 你做了什么 | 每条 bullet 写**你做了什么**，不是项目介绍；体现技能使用，不是学习过程；身份自给，不写「成员」「打杂」 |
| **可阅读性强** | 一眼能读 | 分点是逻辑重构；只用加粗；≤2 种颜色；单栏或非等宽双栏；无口语 |
| **可感知性强** | 一眼能抓重点 | 叙述数字化；标题规整化；词句专业化（不硬凹） |

## 四阶段生成流程

### 阶段 1 · target_model（建岗位画像）

**输入**：JD 原文、岗位名称、公司/项目背景。

**必须在内存里构造一个 JSON 结构，然后继续下一步**：

```yaml
target_model:
  role_type: "校招 | 实习 | 研究申请 | 选调 | 社招"   # 决定骨架走哪种
  function: "技术 | 产品 | 运营 | 研究 | 综合"       # 决定语气和模块排序
  hard_requirements:                                # JD 里明说「必须」「要求」的
    - 硕士及以上
    - Python / PyTorch
    - 英语六级
  soft_preferences:                                  # 「优先」「加分」「熟悉 X 者优先」
    - 有视频图像处理经验
    - 有去噪/超分项目经验
  key_propositions:                                  # 这次申请要证明的 3-5 条核心命题
    - "有真实研究/工程经历，不只上过课"
    - "能快速上手并独立推进任务"
    - "有团队协作与交付能力"
    - "有英语文献阅读和写作能力"
  company_tone: "大厂互联网（量化 / 结果导向）"      # 决定动词和数字密度
  skeleton: "通用校招型"                              # 见下方四种骨架
  do_not_emphasize:                                  # 明确哪些材料这次用不上
    - "与岗位无关的远期校园经历"
    - "学生工作细节（除非在证明团队协作命题）"
```

**自检**：`key_propositions` 每一条是否都能在 JD 里找到依据？找不到说明过度发挥，删掉。

---

### 阶段 2 · evidence_slate（选经历）

**输入**：target_model + canonical profile（所有候选经历的原材料）。

给每条候选经历做一张「证据卡」：

```yaml
- id: "杭州新科技生态探析"
  time: "2024-07 ~ 2024-08"
  role: "调研组长"
  summary_one_line: "7 人团队走访 12 家企业，产出 8000+ 字产业报告"
  supports_propositions:           # 该经历支撑哪些命题
    - "有真实研究/工程经历"
    - "有团队协作与交付能力"
  score:                           # 五维打分，每项 1-5
    relevance: 4                   # 相关度
    recency: 5                     # 时效性（2 年内=5）
    output_salience: 4             # 成果可感知性
    personal_contribution: 4       # 个人贡献清晰度
    uniqueness: 3                  # 不可替代性
  total: 20
  decision: "正文"                 # 正文 / 辅助 / 舍弃
  risk_notes: "企业名可能脱敏，避免造成客户信息泄露"
```

**硬规则**：

- 总分 **< 15** → 舍弃，不进简历
- 总分 **16-20** → 辅助位（自我评价 / 奖项短条 / 技能说明）
- 总分 **> 20** → 正文核心经历
- 正文最终 **2-4 条**，建议 **3 条**（PDF 指南明确标准）
- 至少 **1 条**与 JD 专业技能密切相关（不满足不要生成，回阶段 1 修正 target_model 或跟用户沟通素材不足）

**筛选策略**：如果多条经历支撑同一命题，只留分数最高的一条，其余降为辅助或舍弃——每个命题一条正文证据就够，堆砌会稀释焦点。

---

### 阶段 3 · master_facts（锁事实）

**输入**：canonical profile + evidence_slate。

在写任何文案前，先把**容易漂移的硬事实**抽成单一主口径：

```yaml
master_facts:
  basic:
    name: "张三"
    email: "zhangsan@pku.edu.cn"            # 学校邮箱优先（PDF 明确建议）
    phone: "138xxxxxxxx"
  academics:
    school: "北京大学"                       # 和 JD 不匹配不造假
    major: "信息管理与信息系统"
    degree: "本科"
    gpa: "3.8/4.0"                          # 挑最好看的计算法，太难看不写
    rank: "前 15%"                           # 太靠后不写
    graduation: "2025.06"
  experiences_facts:                         # 每条正文经历的硬事实
    - id: "杭州新科技生态探析"
      time_range: "2024.07 ~ 2024.08"
      role: "调研组长"
      org: "某院系调研项目"
      quantified_outputs:
        - "走访 12 家企业"
        - "7 人团队"
        - "8000+ 字报告"
  exam_scores:
    cet6: "560"
  awards:
    - name: "校级一等奖学金"
      year: ["2022", "2023"]
  target_role_override: null                 # 这次申请如果需要把意向岗位写进简历，这里填
```

**硬规则**（PDF「美化的边界」落地）：

| 绝不能动 | 允许的美化 |
|--------|-----------|
| 学校名（写"北京大学"就是北京大学，不是"北大 XX 学院"） | GPA 选学校承认的最好算法 |
| 学历层次、专业 | 课程重修算最后一次最高分 |
| 第 N 作者、第 N 负责人 | 身份"成员"→"核心成员"（你要真正熟悉整个项目） |
| 时间区间 | "55 份"→"近 60 份"（四舍五入可以，"近百份"过分） |
| 没做过的项目 | "客套几句"→"获高度认可" |
| 获奖等级、名次 | "半个月完成"→"从 0 迅速学习 1 周内基本完成"（有基础时可说） |
| 考试成绩精确分数 | "良率 3%→5%"→"良率近乎翻倍" |

阶段 4 写的每一句都只能引用 master_facts 里的数据。看到要写一个数字、头衔、时间、奖项时，先查 master_facts，**没有就空着、改写或删掉，绝不临场造**。

### 阶段 4 · 写作（先提纲后落笔）

**输入**：target_model + evidence_slate + master_facts。

**第 1 步：生成 resume_outline，不写正文**

```yaml
resume_outline:
  order:                              # 从 target_model.skeleton 和 function 推导
    - basic_info
    - education
    - experiences                     # 如果技术岗可把 skills 提前
    - campus_experience
    - skills
    # - self_evaluation               # 只有内容确实不够充实才加
  section_goals:
    basic_info: "姓名 / 联系方式 / 意向岗位 / 学校 / 专业 / 政治面貌（国企场景）"
    education: "证明硬条件（学历 / 专业）+ 学术证据（GPA / 课程 / 奖学金 1 行）"
    experiences:
      - proposition_proven: "有真实研究/工程经历"
        entry: "杭州新科技生态探析"
        bullets_plan:
          - "背景 + 角色 + 核心动作（调研方法）"
          - "产出（报告字数 + 走访企业数）"
          - "结果（被采纳/引用/内部流转）"
      # ... 2-3 个 entries 同样结构
    skills:
      - "Python / Pandas / NumPy（核心）"
      - "SQL（基础）"
      - "英语六级 560"
```

**第 2 步：每条 bullet 强制回答五件事**

这是 PDF 的通用公式落地。每条 bullet 生成时必须能填满下表五格，不能填的就不写：

| 五要素 | 来源 | 例 |
|-------|------|---|
| 背景 | master_facts 或 evidence.summary | 面对高校间激烈生源竞争 |
| 个人定位 | evidence.role | 作为负责人 |
| 动作 / 方法 | evidence + bullets_plan | 运用信息搜集与沟通交流能力 |
| 工作内容 + 量化 | master_facts.quantified_outputs | 完成线上宣讲近 10 次、答疑逾 200 人次 |
| 成效 | master_facts.quantified_outputs | 吸引 36 位学子，较前 5 年均值 +112% |

缺「成效」是最常见问题。**没结果就写可感知产出**（文档、代码、报告）。**没量化就写可核查完成物**（XX 份、XX 项）。**个人贡献说不清就降级动词**（主导→参与），不要硬凹。

**第 3 步：可读性成型**

- 每段经历 **2-3 整行**，分点 **≤3 点**，每点 **≤2 行**
- 只用加粗一种强调；不混斜体、下划线、高亮
- 单栏 或 非等宽双栏（左窄右宽）；绝不等宽双栏、绝不三栏以上
- 颜色 ≤2 种；主文本黑色，标题/块可以用不刺眼的同色系（如深蓝 #1F3864）
- 本科生 A4 一页；超页先删弱相关经历，最后才考虑字号

## 六项 Review（生成后必跑）

写完 resume_data 后、**渲染 docx 之前**，逐项自检。**任何一项失败必须回阶段 1-3 修正，不得直接渲染**。

### ① 人岗匹配检查

自问一句话：**「这个人为什么适合这个岗位？」** 回答不上来就 fail。

通过标准：简历里应能直接引出 target_model.key_propositions 里至少 3 条的证据。

### ② 核心叙事一致性检查

输出两个东西：

- **本简历的核心标签**（单句）
- **最关键的三条证据**（各一句）

核心标签若和三条证据对不上，或三条证据之间没有共同主线，说明还只是「信息汇总稿」，不是「申请稿」。回阶段 2 重选证据。

### ③ 经历选择正确率检查

正文 3-5 条经历，**每条都要能在 evidence_slate 里找到 score ≥ 20 的原因**。找不到 → 换证据。同时：

- 硬条件（target_model.hard_requirements）是否都在简历能被看到？
- 低分经历是否误入正文？

### ④ Bullet 完整性检查（PDF 六大欠佳模式反查）

对每一条 bullet 做以下问答：

| 反模式 | 症状 | 修复 |
|-------|------|------|
| **人岗不匹配** | 招生工作经历 → 应聘图像算法 | 该经历降级为辅助或舍弃 |
| **扬短避长、过于诚实** | 「熬夜赶进度」「犯了错误」 | 删负面细节，只保留结果 |
| **混乱难以阅读** | 记叙文式「第一天…第二天…」 | 改为动词+名词结构；分点 |
| **说了等于没说** | 「参与准备，发挥能力，取得成效」 | 必须可量化：XX 项 / XX 人次 / +XX% |
| **不说人话** | 「私域建联」「IP 活动」「顶层设计」 | 换具体行为描述 |
| **拖沓 / 复述职责** | 整段讲组长要求、岗位职责 | 删职责通述，只留「我做了什么→结果」 |
| **其他不当表述** | 实习生写「主导战略」「官宣代言人」 | 按实际参与深度选动词 |

### ⑤ 事实准确率检查

**每个数字 / 头衔 / 时间 / 奖项 / 考试成绩**逐项对照 master_facts：

- 有没有引用 master_facts 之外的数据？→ fail
- 有没有和 master_facts 冲突？→ fail
- 学校、学历、专业名是否精确？→ 一个字也不能偏

### ⑥ 可读性检查

- 每段经历长度在 2-3 整行？
- bullet 数 ≤3？单 bullet ≤2 行？
- 加粗量适度（整份简历 ≤15 处）？
- 无口语词（「然后」「就是」「我觉得」）？
- 单栏或非等宽双栏？颜色 ≤2 种？
- 本科生一页以内？

## 隐私与脱敏（默认行为）

`信息.md` 是**信息仓**，可以保留身份证号、详细住址、学号、完整生日。
简历 docx 是**投递品**，默认脱敏：

| 字段 | 信息.md | 简历 docx |
|-----|---------|----------|
| 姓名 | ✓ | ✓ |
| 手机 | ✓ | ✓ |
| 邮箱 | ✓ | ✓（优先学校邮箱） |
| 学校 / 专业 / 学历 | ✓ | ✓ |
| 政治面貌 | ✓ | 国企/选调/高校/事业单位 ✓；互联网/外企默认省略 |
| 意向岗位 | ✓ | **必须** ✓ |
| 性别 | ✓ | 有照片时可省略 |
| 年龄 | ✓ | 社招必写，校招视情况 |
| 籍贯 | ✓ | 选调必写，其他默认省略 |
| **身份证号** | ✓ | **永不**进简历 |
| **详细住址** | ✓ | **永不**进简历 |
| **学号** | ✓ | **永不**进简历 |
| **完整生日** | ✓ | 省略（年龄已覆盖） |

`scripts/field_mapping.py:sanitize_for_resume` 在渲染前自动执行这个策略；LLM 不需要每次手动删。

## 四种骨架

根据 `target_model.skeleton` 选：

### 通用校招型（互联网 / 大厂 / 技术 / 综合管培）

```
基本信息 → 教育背景 → 项目/实习经历 → 校园经历 → 技能特长
```

语气：量化、结果导向；动词偏好主导/构建/优化/落地/迭代；不带照片（互联网）。

### 项目申请型（研究项目、海外交换、特训营）

```
基本信息 → 申请定位（1-3 行动机 + 契合度） → 教育背景 → 实践经历 → 组织与沟通经历 → 荣誉与技能
```

允许"申请定位"段简短自陈，直接对题。动词偏好研究/分析/调研/组织。

### 国企 / 选调 / 事业单位型

```
基本信息（政治面貌前置） → 教育背景 → 学生工作 → 实践经历 → 荣誉 → 技能
```

语气：稳重、严谨、有全局观；动词偏好组织/协调/推动/完成/参与；政治面貌必写；党员靠前；需附照片；籍贯需写。

### 研究 / 学术型

```
基本信息 → 教育背景（GPA/课程/荣誉详写） → 科研项目 → 论文/学术成果 → 技能 → 其他
```

学术成果单独列出；动词偏好研究/分析/设计实验/验证假设；校园活动压缩。

## Agent 执行 Checklist

进入 generate phase 时，**每次都完整走一遍**：

- [ ] 读取 `profile`（canonical）和 JD 原文
- [ ] **阶段 1**：构造 target_model；自检 key_propositions 都来自 JD
- [ ] **阶段 2**：构造 evidence_slate；五维打分；正文选 2-4 条；至少 1 条对接专业硬条件
- [ ] **阶段 3**：提取 master_facts；再检查没有虚构字段
- [ ] **阶段 4.1**：构造 resume_outline；确定骨架 / 模块顺序 / 每段 bullets_plan
- [ ] **阶段 4.2**：每条 bullet 填满五要素公式，缺「成效」降级或删
- [ ] **阶段 4.3**：检查长度 / 加粗 / 分点 / 颜色
- [ ] 调 `sanitize_for_resume(profile, target_role)` 脱敏
- [ ] 六项 review 全部通过
- [ ] 调 `render_resume.py`，preflight 若返回 `preflight_problems` 非空，回上一步修
- [ ] 输出结果 + 写 version metadata（`.resume/resumes/<ts>/meta.json`）

**未通过 review → 不渲染。**
